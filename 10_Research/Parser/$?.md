Puisque **`$?`** est une obligation absolue pour réussir le minishell (et c'est souvent là que l'[[_Expander|Expander]] bugge), on va faire une petite docu pour comprendre à quoi sa sert !

# Le `$?` (Le Code de Retour / Exit Status)

### 1. Qu'est-ce que c'est ?

Imagine que chaque commande qu'on lance (ex: `ls`, `cat`, ou un de tes [[_Builtins|builtins]]) est un soldat qu'on envoi en mission. Quand le soldat a fini, il revient te voir et te donne un **rapport de mission** sous la forme d'un simple numéro.

Ce numéro, c'est le **code de retour** (Exit Status). Le symbole **`$?`** est la "boîte" magique dans laquelle le Shell range automatiquement le tout dernier rapport reçu.

### 2. Mais le code, c'est pas le code ?

Contrairement au C où `1` signifie souvent "Vrai/Succès" et `0` "Faux", dans Bash c'est l'inverse :

- **`0` = SUCCÈS TOTAL.** (Tout s'est bien passé).

- **`1` à `255` = ERREUR.** (Plus le chiffre est précis, plus on sait ce qui a foiré).


### 3. Les Codes d'Erreur Classiques

Notre minishell doit absolument respecter les mêmes codes d'erreur que le vrai Bash, ceux sur lesquels on va probablement devoir gérer seront ceux-là :

| **Code ($?)** | **Signification**                         | **Exemple d'utilisation**                                                                                  |
| ------------- | ----------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| **`0`**       | Succès                                    | `echo "bonjour"` ou `ls` (si le dossier existe)                                                            |
| **`1`**       | Erreur générale                           | `ls /dossier_qui_nexiste_pas` ou un `cd` raté                                                              |
| **`2`**       | Erreur de syntaxe (Bash)                  | Oubli de fermer des guillemets, ou erreur de Lexer                                                         |
| **`126`**     | Commande trouvée, mais **non exécutable** | On essai de lancer un dossier au lieu d'un fichier (`./mon_dossier`) ou on n'a pas les droits (`chmod -x`) |
| **`127`**     | Commande **introuvable**                  | On tape `salut_ca_va` (qui n'existe pas dans le `$PATH`)                                                   |
| **`130`**     | Interrompu par `Ctrl-C`                   | L'utilisateur a tué la commande en cours avec `SIGINT` (128 + signal 2)                                    |
| **`131`**     | Interrompu par `Ctrl-\`                   | L'utilisateur a quitté violemment avec `SIGQUIT` (128 + signal 3)                                          |

**La formule magique des signaux :** Quand un processus est tué par un signal, Bash calcule toujours son `$?` en faisant **`128 + numéro_du_signal`**. C'est pour ça que `Ctrl-C` (qui est le signal n°2) donne `$? = 130`.

### 4. Comment ça marche dans l'architecture de notre Minishell ?

Le `$?` fait le pont entre la fin de notre programme (l'[[_Executor|Executor]]) et le début (l'[[_Expander|Expander]]).

#### A. Côté Executor

Quand ton [[_Executor|Executor]] fait un `waitpid()` pour attendre la fin d'une commande enfant, il récupère un `status`. C'est là qu'on met à jour notre variable globale.

```
int status;
waitpid(pid, &status, 0);

if (WIFEXITED(status))
    g_status = WEXITSTATUS(status);
else if (WIFSIGNALED(status))
    g_status = 128 + WTERMSIG(status);
```

#### B. Côté Expander

Quand l'utilisateur tape : `echo "Le dernier code est $?"`

Ton [[_Expander|Expander]], qui est habitué à chercher des mots dans l'environnement (`$USER`, `$PATH`), doit avoir une règle spéciale, un **"If" exclusif** pour le point d'interrogation.

```
if (nom_de_la_variable == '?')
{
    char *status_str;
    
    status_str = ft_itoa(g_status);
    ligne_finale = remplacer_str(ligne_originale, "$?", status_str);
    free(status_str);
}
```
